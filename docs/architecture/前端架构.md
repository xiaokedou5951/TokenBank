# TokenBank 前端架构

## 1. 目标

说明 TokenBank 前端的目录结构、核心模块职责、数据流以及关键设计决策。

## 2. 关键结论

- 前端基于 **Next.js 16 App Router + React 19 + Tailwind CSS 4** 构建，开发服务器使用 Turbopack。
- Web3 交互统一通过 **Wagmi v2 + Viem v2** 完成，钱包连接由 **RainbowKit** 提供。
- 状态管理以 **TanStack Query（React Query）** 为核心，Wagmi Hooks 已内置缓存与刷新能力。
- 合约 ABI 采用**内联 `as const` 片段**管理，避免额外生成步骤，仅包含前端实际使用的函数、事件和错误。
- 存款支持两种模式：**传统 Approve+Deposit**（两步）和 **Permit Deposit**（EIP-2612 签名，一步完成）。

## 4. 核心模块职责

### 4.1 配置层

| 文件 | 职责 |
|------|------|
| `lib/wagmi.ts` | 定义本地 Anvil 链，配置 RainbowKit + Wagmi 支持的链与 RPC transport |
| `lib/contracts.ts` | 定义 `myTokenAbi`（含 Permit/nonces/DOMAIN_SEPARATOR）/ `tokenBankAbi`（含 permitDeposit）内联 ABI 片段，管理合约地址 |
| `app/providers.tsx` | 组合 RainbowKit、WagmiConfig、QueryClientProvider |

### 4.2 Hooks 层

#### useToken.ts

| Hook | 说明 |
|------|------|
| `useTokenBalance(address)` | 读取用户钱包中的 MyTokenPermit 余额 |
| `useTokenAllowance(address)` | 读取用户对 TokenBankPermit 的授权额度 |
| `useTokenNonce(address)` | 读取用户 EIP-2612 nonce（用于 Permit 签名） |
| `useTokenDomainSeparator()` | 读取代币合约的 DOMAIN_SEPARATOR |
| `useApprove()` | 调用 `approve(TokenBankPermit, amount)`，交易成功后刷新相关查询 |
| `usePermitSignature()` | 生成 EIP-2612 Permit 签名（`signTypedDataAsync`），返回签名结果 |

#### useTokenBank.ts

| Hook | 说明 |
|------|------|
| `useDepositBalance(address)` | 读取用户在 TokenBankPermit 中的存款余额 |
| `useDeposit()` | 调用 `TokenBankPermit.deposit(amount)`（传统方式） |
| `usePermitDeposit()` | 调用 `TokenBankPermit.permitDeposit(amount, deadline, v, r, s)` |
| `useWithdraw()` | 调用 `TokenBankPermit.withdraw(amount)` |

所有写操作 Hook 统一返回 `isPending`（等待签名）、`isConfirming`（等待上链）、`isSuccess`（成功）、`error`（错误）状态，供 UI 展示。

### 4.3 组件层

| 组件 | 职责 |
|------|------|
| `ConnectWallet` | 触发 RainbowKit 连接弹窗 |
| `BalanceCard` | 聚合展示三类余额：钱包 Token 余额、Bank 存款余额、对 Bank 的授权额度 |
| `DepositForm` | 输入金额，提供三种操作：Approve（传统授权）、Deposit（传统存款）、Permit Deposit（一键签名存款） |
| `WithdrawForm` | 输入金额，从 Bank 取款 |
| `ActivityLog` | 展示近期交易 hash、状态、成功/失败信息 |

页面布局采用响应式网格：左侧 8/12 为主内容（余额 + 表单），右侧 4/12 为交易记录。

## 5. 数据流

### 5.1 通用读写模式

```
用户操作 → React 组件 → 自定义 Hook → Wagmi / Viem → JSON-RPC → 智能合约
                ↑                                    │
                └──── React Query 缓存与刷新 ←────────┘
```

- **读数据**：`useReadContract` 配合 `enabled: !!address`，未连接钱包时不发起请求。
- **写数据**：`useWriteContract` 发起交易，`useWaitForTransactionReceipt` 监听回执。
- **刷新**：交易成功后调用 `queryClient.invalidateQueries({ queryKey: ['readContract'] })`，统一刷新余额与授权。

### 5.2 Permit Deposit 数据流

```
用户点击 "Permit Deposit"
  │
  ▼
DepositForm.handlePermitDeposit()
  │ 1. 读取 nonce（useTokenNonce）
  │ 2. 计算 deadline = 当前时间 + 1小时
  │ 3. 调用 usePermitSignature().generatePermitSignature()
  ▼
钱包弹出 EIP-712 签名请求
  │ 用户签名 → 返回 signature (hex 65 bytes)
  ▼
signature useEffect 触发
  │ 4. 从 signature 提取 v, r, s
  │ 5. 调用 usePermitDeposit().permitDeposit(amount, deadline, v, r, s)
  ▼
TokenBankPermit.permitDeposit() 链上执行
  │ 6. permit(owner, spender, value, deadline, v, r, s)
  │ 7. safeTransferFrom(user, bank, amount)
  │ 8. emit PermitDeposit(user, amount)
  ▼
交易确认后 invalidateQueries → 刷新余额
```

## 6. 关键设计决策

| 决策 | 选择 | 原因 |
|------|------|------|
| ABI 管理方式 | 内联 `as const` 片段 | 直接为 Wagmi/Viem 提供类型推断，无需额外类型生成步骤 |
| 合约地址管理 | 环境变量 + Anvil fallback | 支持多环境，本地开发可零配置启动 |
| 状态管理 | React Query（Wagmi 内置） | 无需 Redux 等全局状态库，缓存、重试、刷新已覆盖需求 |
| 组件粒度 | BalanceCard 聚合三类余额 | 减少组件数量，当前页面信息密度低，聚合展示更直观 |
| 链配置 | Anvil + Sepolia + Mainnet | 本地开发、测试网、生产预留 |
| Permit 签名 | `signTypedDataAsync` | wagmi v2 中 `signTypedData` 不返回 Promise，需用 async 版本等待签名完成 |
| 签名→提交 | useEffect 监听 signature | 签名完成后自动触发 permitDeposit，用户无需二次操作 |
| nonce 检查 | `nonce === undefined` 而非 `!nonce` | 首次交易 nonce 为 `0n`，falsy 检查会误判为未加载 |
| deadline 存储 | state 变量 `permitDeadline` | 签名与提交时使用同一 deadline，避免两次 `Date.now()` 不一致 |
