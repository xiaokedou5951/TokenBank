# TokenBank 架构文档组织方式

## 1. 设计原则

针对 TokenBank 当前规模（2 个合约 + 1 个 Next.js 前端），文档组织遵循 **"少即是多、按需演进"** 的原则：

- **能合并则合并**：相近主题合为一份文档，避免碎片化和过度分类。
- **先写必要，后补扩展**：当前只记录理解系统所必需的内容，不预建空文档。
- **代码即文档**：合约内已写清注释的函数行为、前端 hooks 内的类型定义，不在文档中重复誊抄。
- **一份入口导航**：通过 `docs/README.md` 统一指引读者，避免在目录中迷路。

## 2. 目录结构

```
docs/
├── README.md                          # 文档入口与阅读指南
├── prd/
│   └── TokenBank需求.md               # 产品需求（已存在）
├── architecture/                      # 架构设计文档
│   ├── README.md                      # 架构文档入口
│   ├── 架构文档组织方式.md             # 本文档：说明如何组织架构文档
│   ├── 系统架构概览.md                 # 系统整体分层、模块关系、部署视图
│   ├── 智能合约架构.md                 # 合约职责、交互时序、安全考量
│   └── 前端架构.md                     # 前端目录结构、数据流、技术决策
├── dev/                               # 开发运维文档
│   ├── 环境搭建.md                     # 本地开发环境、依赖安装、启动步骤
│   └── 部署指南.md                     # 合约部署、前端部署、环境变量说明
└── decisions/
    └── 技术决策记录.md                 # 关键决策及原因，按时间顺序记录
```

## 3. 各文档职责

| 文档 | 职责 | 读者 | 更新时机 |
|------|------|------|----------|
| `docs/README.md` | 说明文档结构、阅读顺序、最新版本约定 | 新加入开发者、审计者 | 目录调整时 |
| `architecture/README.md` | 简述系统目标、技术栈、核心模块一览，并链接到各子文档 | 所有读者 | 架构大调整时 |
| `架构文档组织方式.md` | 规定架构文档如何分类、命名、演进 | 文档维护者 | 组织方式变化时 |
| `系统架构概览.md` | 系统分层（链上 / 前端 / 外部依赖）、模块边界、部署拓扑 | 新加入开发者、评审者 | 新增模块或部署环境时 |
| `智能合约架构.md` | 合约清单、核心函数、调用时序、安全机制、风险点 | 合约开发者、审计者 | 合约升级或新增功能时 |
| `前端架构.md` | 前端目录约定、组件与 hooks 职责、数据流、ABI 管理 | 前端开发者 | 前端结构变化时 |
| `dev/环境搭建.md` | 本地环境准备、合约编译测试、前端启动 | 新加入开发者 | 依赖或命令变化时 |
| `dev/部署指南.md` | 合约部署脚本说明、前端环境变量、部署检查清单 | 部署者 | 部署流程变化时 |
| `decisions/技术决策记录.md` | 记录重大技术选择及原因，如内联 ABI、RainbowKit 等 | 长期维护者 | 做新决策时 |

## 4. 内容规范

### 4.1 标题与命名

- 文件名使用中文，简洁表达主题，如 `智能合约架构.md`。
- 文档内一级标题与文件名保持一致。
- 新文档创建前先在 `docs/README.md` 中登记。

### 4.2 避免的内容

- 不复制代码注释：函数级说明应留在源码中。
- 不写与代码不一致的流程：若代码更新，文档应同步或标注过期。
- 不记录临时调试命令：命令类内容放在 `dev/环境搭建.md` 或脚本注释中。

### 4.3 建议包含的结构

每份架构文档建议按以下结构编写：

```markdown
# 标题

## 1. 目标
说明本文档要回答什么问题。

## 2. 关键结论
用 3-5 条 bullet 列出最重要的设计结论。

## 3. 详细说明
必要时补充图表、时序、目录结构。

## 4. 待决策 / 待办
记录当前尚未确定或需要后续跟进的事项。
```

## 5. 演进规则

1. **当前规模**：保持 `docs/` 目录在 10 份文档以内，优先复用已有文档。
2. **规模扩大时**：
   - 若单一文档超过 400 行，可考虑拆分子主题。
   - 若新增独立模块（如后端服务、索引器），新增对应架构文档。
3. **归档机制**：
   - 过时的设计说明迁移到 `decisions/` 并标注过期时间，不直接删除。
   - 实验性内容可新建 `docs/experiments/` 子目录，成熟后再纳入正式文档。
