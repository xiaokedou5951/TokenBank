# TokenBank 智能合约架构

## 1. 目标

说明 TokenBank 链上合约的职责划分、核心交互时序、安全机制及潜在风险。

## 2. 关键结论

- 共 2 个合约：`MyTokenPermit`（支持 EIP-2612 的 ERC20 代币）与 `TokenBankPermit`（支持 Permit 存款的银行合约）。
- `MyTokenPermit` 继承 OpenZeppelin 的 `ERC20Permit`，实现 EIP-2612 签名授权，支持离线 approve。
- `TokenBankPermit` 在构造时绑定底层代币地址，提供传统 `deposit` 和 `permitDeposit` 两种存款方式。
- 取款遵循 Checks-Effects-Interactions 模式，配合 `ReentrancyGuard` 防范重入。
- `permitDeposit` 将 permit + transferFrom 合并为单笔交易，用户无需提前发起 approve 交易。

## 3. 合约清单

| 合约 | 文件 | 核心职责 | 关键依赖 |
|------|------|----------|----------|
| MyTokenPermit | `contracts/src/MyTokenPermit.sol` | ERC20 代币，支持铸造、销毁、EIP-2612 Permit | OpenZeppelin ERC20 / ERC20Permit / Burnable / Ownable |
| TokenBankPermit | `contracts/src/TokenBankPermit.sol` | 接收、记录并返还用户存入的代币，支持 Permit 存款 | OpenZeppelin IERC20 / IERC20Permit / SafeERC20 / ReentrancyGuard |

## 4. 状态与函数

### 4.1 MyTokenPermit

- 构造时向部署者铸造全部初始供应量（`initialSupply * 10^18`）。
- `decimals()` 固定返回 18。
- `mint(to, amount)` 仅 owner 可调用。
- 继承 ERC20Permit，自动提供 `permit()`、`nonces()`、`DOMAIN_SEPARATOR()` 方法。

### 4.2 TokenBankPermit

| 类型 | 名称 | 说明 |
|------|------|------|
| 状态 | `token` | 不可变的底层 ERC20 地址 |
| 状态 | `balances[address]` | 用户在 Bank 中的存款余额 |
| 写函数 | `deposit(uint256 amount)` | 传统存款：从用户转入代币并增加存款余额（需提前 approve） |
| 写函数 | `permitDeposit(uint256 amount, uint256 deadline, uint8 v, bytes32 r, bytes32 s)` | Permit 存款：先调用 permit 授权再转入代币，单笔交易完成 |
| 写函数 | `withdraw(uint256 amount)` | 校验余额后返还代币 |
| 读函数 | `balanceOf(address)` | 查询用户存款余额 |
| 事件 | `Deposit(address user, uint256 amount)` | 传统存款事件 |
| 事件 | `PermitDeposit(address user, uint256 amount)` | Permit 存款事件 |
| 事件 | `Withdraw(address user, uint256 amount)` | 取款事件 |
| 错误 | `ZeroAmount` | 金额为 0 |
| 错误 | `ZeroAddress` | 代币地址为 0 |
| 错误 | `InsufficientBalance` | 余额不足 |
| 错误 | `PermitFailed` | Permit 调用失败 |

## 5. 调用时序

### 5.1 传统存款时序（Approve + Deposit）

```
用户
  │ 1. approve(TokenBankPermit, amount)
  ▼
MyTokenPermit
  │ 授权 TokenBankPermit 可动用用户代币
  ▼
用户
  │ 2. deposit(amount)
  ▼
TokenBankPermit
  │ 调用 token.safeTransferFrom(user, TokenBankPermit, amount)
  │ balances[user] += amount
  │ emit Deposit(user, amount)
```

### 5.2 Permit 存款时序（一键签名存款）

```
用户（链下）
  │ 1. 签名 EIP-712 typed data（Permit 结构体）
  │    domain = { name, version, chainId, verifyingContract }
  │    message = { owner, spender, value, nonce, deadline }
  ▼
用户（链上）
  │ 2. permitDeposit(amount, deadline, v, r, s)
  ▼
TokenBankPermit
  │ 调用 IERC20Permit(token).permit(owner, TokenBankPermit, amount, deadline, v, r, s)
  │   → MyTokenPermit.permit() 验证签名并更新 allowance
  │ 调用 token.safeTransferFrom(user, TokenBankPermit, amount)
  │ balances[user] += amount
  │ emit PermitDeposit(user, amount)
```

### 5.3 取款时序

```
用户
  │ withdraw(amount)
  ▼
TokenBankPermit
  │ require(amount > 0)
  │ require(balances[user] >= amount)
  │ balances[user] -= amount          // 先改状态
  │ token.safeTransfer(user, amount)  // 再转账
  │ emit Withdraw(user, amount)
```

## 6. 安全机制

| 机制 | 实现位置 | 作用 |
|------|----------|------|
| 零地址检查 | `TokenBankPermit.constructor` | 防止绑定无效代币地址 |
| 金额检查 | `TokenBankPermit.deposit/permitDeposit/withdraw` | 拒绝 0 金额或超额取款 |
| 重入防护 | `TokenBankPermit` 继承 `ReentrancyGuard` | 防止存款/取款/permitDeposit 重入攻击 |
| CEI 模式 | `TokenBankPermit.withdraw` | 先更新状态再转账，降低重入风险 |
| 安全转账 | `SafeERC20.safeTransferFrom` / `safeTransfer` | 兼容非标准 ERC20 并实现调用失败回滚 |
| Permit 验证 | `ERC20Permit.permit`（OpenZeppelin） | EIP-712 签名验证，含 nonce 防重放、deadline 防过期 |
| try/catch 保护 | `TokenBankPermit.permitDeposit` | permit 调用失败时回滚为自定义 `PermitFailed` 错误 |

## 7. EIP-2612 Permit 详解

### 7.1 核心概念

EIP-2612 允许用户通过**链下签名**代替链上 `approve` 交易，实现 gasless approval：

- **签名结构**：用户对 EIP-712 typed data 签名，包含 `owner`、`spender`、`value`、`nonce`、`deadline`。
- **nonce 机制**：每个地址维护一个 nonce，防止签名重放。
- **deadline**：签名有效期，过期后 permit 调用失败。
- **v/r/s**：ECDSA 签名的三个分量，由前端从钱包签名结果中提取。

### 7.2 与传统方式的对比

| 对比项 | 传统 Approve + Deposit | Permit Deposit |
|--------|----------------------|----------------|
| 交易数 | 2 笔（approve + deposit） | 1 笔（permitDeposit） |
| 用户操作 | 签名 2 次 | 签名 1 次（链下 EIP-712） |
| Gas 费用 | 更高（两笔交易） | 更低（一笔交易，permit + transferFrom） |
| 前置条件 | 需要先发起 approve 交易 | 无需提前操作 |

## 8. 风险与注意事项

- **代币风险**：TokenBankPermit 假设底层代币符合 ERC20 标准，不包含 rebase 代币、转账税代币等特殊逻辑处理。
- **权限风险**：当前 TokenBankPermit 无 owner、无 pause、无紧急提现，部署后不可变更。
- **Permit 签名风险**：用户签名后，任何人在 deadline 前都可提交该签名；前端应确保签名仅发送到目标合约。
- **小数精度**：合约层面使用原始 uint256；前端需按 `decimals = 18` 进行格式化与解析。
