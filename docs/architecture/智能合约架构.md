# TokenBank 智能合约架构

## 1. 目标

说明 TokenBank 链上合约的职责划分、核心交互时序、安全机制及潜在风险。

## 2. 关键结论

- 共 2 个合约：`MyToken`（标准 ERC20）与 `TokenBank`（存取逻辑）。
- `TokenBank` 在构造时绑定底层代币地址，本身无管理权限、无升级逻辑。
- 存款采用标准的 `approve` + `transferFrom` 模式。
- 取款遵循 Checks-Effects-Interactions 模式，配合 `ReentrancyGuard` 防范重入。

## 3. 合约清单

| 合约 | 文件 | 核心职责 | 关键依赖 |
|------|------|----------|----------|
| MyToken | `contracts/src/MyToken.sol` | ERC20 代币，支持铸造、销毁、Permit | OpenZeppelin ERC20 / Burnable / Permit / Ownable |
| TokenBank | `contracts/src/TokenBank.sol` | 接收、记录并返还用户存入的代币 | OpenZeppelin IERC20 / SafeERC20 / ReentrancyGuard |

## 4. 状态与函数

### 4.1 MyToken

- 构造时向部署者铸造全部初始供应量（`initialSupply * 10^18`）。
- `decimals()` 固定返回 18。
- `mint(to, amount)` 仅 owner 可调用。

### 4.2 TokenBank

| 类型 | 名称 | 说明 |
|------|------|------|
| 状态 | `token` | 不可变的底层 ERC20 地址 |
| 状态 | `balances[address]` | 用户在 Bank 中的存款余额 |
| 写函数 | `deposit(uint256 amount)` | 从用户转入代币并增加存款余额 |
| 写函数 | `withdraw(uint256 amount)` | 校验余额后返还代币 |
| 读函数 | `balanceOf(address)` | 查询用户存款余额 |
| 读函数 | `totalDeposits()` | 查询 Bank 合约持有的代币总量 |
| 事件 | `Deposit / Withdraw` | 记录存取行为 |
| 错误 | `ZeroAmount / InsufficientBalance` | 参数与余额校验 |

## 5. 调用时序

### 5.1 存款时序

```
用户
  │ 1. approve(TokenBank, amount)
  ▼
MyToken
  │ 授权 TokenBank 可动用用户代币
  ▼
用户
  │ 2. deposit(amount)
  ▼
TokenBank
  │ 调用 token.safeTransferFrom(user, TokenBank, amount)
  │ balances[user] += amount
  │ emit Deposit(user, amount)
```

### 5.2 取款时序

```
用户
  │ withdraw(amount)
  ▼
TokenBank
  │ require(amount > 0)
  │ require(balances[user] >= amount)
  │ balances[user] -= amount          // 先改状态
  │ token.safeTransfer(user, amount)  // 再转账
  │ emit Withdraw(user, amount)
```

## 6. 安全机制

| 机制 | 实现位置 | 作用 |
|------|----------|------|
| 零地址检查 | `TokenBank.constructor` | 防止绑定无效代币地址 |
| 金额检查 | `TokenBank.deposit/withdraw` | 拒绝 0 金额或超额取款 |
| 重入防护 | `TokenBank` 继承 `ReentrancyGuard` | 防止存款/取款重入攻击 |
| CEI 模式 | `TokenBank.withdraw` | 先更新状态再转账，降低重入风险 |
| 安全转账 | `SafeERC20.safeTransferFrom` / `safeTransfer` | 兼容非标准 ERC20 并实现调用失败回滚 |

## 7. 风险与注意事项

- **代币风险**：TokenBank 假设底层代币符合 ERC20 标准，不包含 rebase 代币、转账税代币等特殊逻辑处理。
- **权限风险**：当前 TokenBank 无 owner、无 pause、无紧急提现，部署后不可变更。
- **授权风险**：用户需主动授权 TokenBank 花费其代币；前端应提示授权额度。
- **小数精度**：合约层面使用原始 uint256；前端需按 `decimals = 18` 进行格式化与解析。
